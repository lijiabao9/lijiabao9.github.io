Ubuntu 服务器 Java 环境部署指南

发表于2026-05-24|更新于2026-05-24|开发环境Ubuntu

在 Ubuntu 服务器上部署 Java 环境通常有两种方式：

  1. 包管理器安装 (apt)：最简单快捷，通常安装的是 OpenJDK，适合绝大多数开源项目与生产环境。
  2. 手动解压安装：适合需要特定商业版本（如 Oracle JDK）或需精确控制安装路径的场景。

方案一：通过 apt 自动安装 (推荐)

这是最省心的方式，Ubuntu 官方源会自动处理依赖、安全更新以及环境变量。

1. 安装与验证

在安装之前，建议先更新本地软件包列表，确保获取到最新版本。

2. 管理多个 Java 版本

如果你的系统同时安装了多个版本的 JDK（例如 JDK 8、11 和 17），可以使用 update-alternatives 工具在不同版本间快速切换，无需手动修改环境变量。

补充：Linux 环境变量文件加载顺序

理解配置文件的加载机制，能帮你更精准地决定将变量写在哪里，避免环境污染或自动化脚本失效。

Linux 的环境变量文件本质上是由 影响范围（全局/用户态） 和 生效时机（登录/非登录） 交叉决定的：

💡 .bashrc 与 .profile 的本质区别

  • ~/.profile（一次性加载）：只在用户刚登录系统时执行一次。由于它生命周期只有一次，最适合用来声明个人态的 JAVA_HOME 等核心环境变量。
  • ~/.bashrc（频繁重复加载）：只要打开新终端窗口或执行自动化脚本，它就会被刷一次。适合存放命令别名（alias）或需要高频实时生效的配置。

⚠️ 为什么必须注意 /etc/ 全局设置？

写在 ~/.profile 或 ~/.bashrc 里的变量无法污染他人，但也无法惠及他人。

  • 生产服务器部署：强烈建议修改全局的 /etc/profile 或 /etc/bash.bashrc，确保系统服务（如 Tomcat、Nginx 调用的后端）以及其他运维账号（如 Jenkins、www）都能全局共享同一套 Java 环境。
  • 自动化脚本避坑：Jenkins 或 Crontab 脚本属于 Non-login Shell，它们会直接跳过 /etc/profile 去读取 ~/.bashrc。因此，最稳妥的做法是在 /etc/profile 配置完后，在全局的 /etc/bash.bashrc 末尾也追加一份，或者在脚本开头显式加上 source /etc/profile。
