二、 故障排查与 Cloudflare 避坑指南

在实际部署中，特别是当你的服务器使用了 Cloudflare (CF) 代理（开启小黄云） 时，可能会遇到各种网络或权限问题。

以下是常见的坑点及完美解决方案：

1. 跨域问题 (CORS 报错)

  • 现象：按 F12 打开浏览器控制台，看到红色报错：No 'Access-Control-Allow-Origin' header is present on the requested resource.
  • 原因：Artalk 出于安全考虑，默认开启了跨域限制，拒绝未授权的域名请求其 API。
  • 解决：
    1. 登录你的 Artalk 后台管理面板。
    2. 进入 设置 -> 可信域名。
    3. 将你的博客域名（例如 https://yourdomain.com）以及本地调试地址 http://localhost:4000 一并添加进去。

2. 混合内容错误 (Mixed Content)

  • 现象：博客启用了 HTTPS（例如部署在 GitHub Pages），Artalk 服务启用的是 HTTP，控制台报错 Mixed Content: The page at '...' was loaded over HTTPS, but requested an insecure resource...
  • 原因：安全策略限制，不允许在 HTTPS 加密页面中请求 HTTP 的明文资源。
  • 解决：检查 _config.butterfly.yml 中的 artalk.server 地址，确保其以 https:// 开头，且你的 Nginx 已经配置好了 SSL 证书。

3. Cloudflare 特有深坑

如果你为 Artalk 的后端域名开启了 Cloudflare 的 CDN 代理（小黄云模式），请务必检查并修改以下四项设置：

⚠️ 坑点 A：端口访问受限

  • 原因：Cloudflare 默认仅代理特定的标准端口（如 HTTP 的 80, 8080，HTTPS 的 443, 8443, 2053 等）。如果你在前端配置的 Server URL 直接使用了 20002 这种自定义端口，CF 会直接拦截。
  • 解决：强烈建议使用 Nginx 反向代理。 让 Artalk 运行在本地 127.0.0.1:20002，通过 Nginx 监听标准的 443 端口并配置 SSL，前端统一请求 https://artalk.yourdomain.com。

⚠️ 坑点 B：后台控制台无法加载 (X-Frame-Options 拦截)

  • 原因：Cloudflare 的安全规则可能会强制为所有请求添加 X-Frame-Options: SAMEORIGIN 响应头，导致 Artalk 后台的 iframe 弹窗因为跨域被浏览器拒绝渲染。
  • 解决：
    1. 登录 Cloudflare 后台 -> 选中对应域名。
    2. 点击左侧菜单的 规则 (Rules) -> 转换规则 (Transform Rules) -> HTTP 响应头 (HTTP Response Headers)。
    3. 检查是否有开启诸如「添加安全性标头」的规则，如有，请针对 Artalk 域名关闭该规则，或者单独放行。

⚠️ 坑点 C：IP 属地无法显示或显示为 Cloudflare 节点

  • 原因：经过 CDN 代理后，Artalk 接收到的直接请求方变成了 Cloudflare 的服务器，从而隐藏了读者的真实 IP。
  • 解决：
    1. CF 侧调整：在 CF 后台 -> 规则 -> 转换规则 -> HTTP 请求头 中，确保没有开启 删除访问者 IP 标头（即保留 CF-Connecting-IP 或 X-Forwarded-For）。
    2. Artalk 侧调整：登录 Artalk 后台 -> 设置 -> 服务器 -> 代理标头名，将其修改为 X-Forwarded-For 或 CF-Connecting-IP。这样 Artalk 就能精准识别并显示读者的地理位置了。

⚠️ 坑点 D：图片上传失败 (403 Forbidden)

  • 原因：读者在评论区上传图片时，触发了 Cloudflare 的防盗链机制，导致请求被直接拦截。
  • 解决：
    1. 登录 Cloudflare 后台 -> 进入 Scrape Shield 设置页。
    2. 找到 Hotlink 保护 (Hotlink Protection) 选项，将其调整为 关闭 状态。
