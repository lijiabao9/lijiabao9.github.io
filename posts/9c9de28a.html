Artalk私有化部署指南：打造轻量、安全的评论系统

发表于2025-06-13|更新于2025-06-23|项目工坊极客玩具
|总字数:3.4k|阅读时长:12分钟|浏览量:

还在依赖 Disqus 或 Valine？Artalk 是一款开箱即用的自托管评论系统，支持Markdown、多站点管理、防垃圾过滤。
本教程将带你从零开始完成 Artalk 的私有化部署，包括 Docker 配置、Nginx反向代理、邮件通知设置，打造完全掌控的评论服务。

环境说明：

  • 操作系统：Ubuntu 22.04 LTS
  • CPU架构：ARMv8（aarch64/arm64）
  • 域名托管到 CloudFlare

参考官文文档：Artalk

基于Docker部署

注意，以上配置中：

  • -p 宿主机端口 23366 按需修改
  • -v 数据卷挂载，宿主机目录 /data/artalk/data 按需修改
  • ATK_SITE_DEFAULT，默认站点的名称，名称随便起就是一个标识符（集成到博客网站需用到）
  • ATK_SITE_URL，默认站点的地址，配置为 GitHub Page 的地址

建议更新容器的 Restart策略 以达到守护进程效果：

容器创建成功后，需要设置管理员账号：


按照提示，输入 昵称、邮箱（登录账号）、密码
例如：admin，admin@example.com，123456

放行服务器端口 23366，浏览器访问地址 http://ip:23366，就是 Artalk 的后台管理系统。
输入 邮箱、密码，登录成功

  • 管理所有人的评论
  • 管理注册的用户
  • 管理站点信息
  • 评论数据迁移
  • Artalk功能设置

修改配置后，可能需要重启服务才会生效。

集成到 Hexo 博客的问题记录

服务部署成功后，集成到 Hexo 是很简单的。
Hexo 安装主题 Butterfly， 对 Artalk 开箱即用。
只需填写 Artalk服务地址（http://ip:port) 和 站点名称(Gabriel的博客)。

但实际上遇到很多坑，记录一下。。。

本地开发环境，加载不出（可信域名）

文章底部，评论区域报错：
Artalk Error
Failed to load comments
TypeError: Failed to fetch
Retry

F12打开控制台，报错：
Access to fetch at ‘http://ip:port/api/v2/conf‘ from origin ‘http://localhost:4000‘ has been blocked by CORS policy:
No ‘Access-Control-Allow-Origin’ header is present on the requested resource.

这是由于 Artalk 服务默认策略，对所有请求源不信任（除了配置的站点地址）。

到 Artalk 后台管理系统，设置中，找到 可信域名 添加上信任的地址，比如开发环境的地址 http://localhost:4000 即可。

正常访问后，文章底部就会出现评论模块功能。
游客需要填写 昵称、邮箱 后，才可留言评论，右侧会有 通知中心，对自己的评论快捷操作。
如果填入的是管理员的昵称和邮箱，会弹出密码验证。成功后，右侧会有 控制中心。是 Artalk 后台管理的简洁模式。

部署到 GitHub Page，加载不出（HTTP协议冲突）

文章底部，评论区域报错：
Artalk Error
Failed to load comments
TypeError: Failed to fetch
Retry

F12打开控制台，报错：
Mixed Content: The page at ‘https://lijiabao9.github.io/posts/5071adbc.html‘ was loaded over HTTPS,
but requested an insecure resource ‘http://ip:port/api/v2/conf‘.
This request has been blocked; the content must be served over HTTPS.

这是由于 协议不一致 导致的。
GitHub Page 站点，默认是使用 HTTPS 协议的。
私有部署的 Artalk 服务默认是 HTTP 协议，跨协议报错了。
本地环境是 HTTP 协议，所以不会报错。

解决方案：将 Artalk 服务升级为 HTTPS 服务。

升级 Artalk 为 HTTPS 服务

参考官方文档 反向代理

前提要拥有 域名 和 证书。
通过 Nginx 做反向 SSL 代理，升级为 HTTPS 协议。

假设域名 artalk.your_domain.com
服务器上新建文件 /etc/nginx/sites-available/artalk.your_domain.com

创建配置文件的软链接

重启 Nginx 服务

现在就可以通过 HTTPS + 域名 访问 Artalk 后台界面了。
比如：https://artalk.your_domain.com

域名托管到 CloudFlare 导致的问题

如果是将域名托管给了 CF，并且开启了代理（橙云）。
可能会出现以下问题（都是踩过的坑。。。）：

  1. 通过域名访问不到后台地址 https://artalk.your_domain:23366，但通过 源服务器的IP 可以访问。
    这是由于 CF 的代理，只支持部分常用的 HTTP/HTTPS 端口，其他端口的流量会直接丢弃。
    HTTP：80, 8080, 8880, 2052, 2082, 2086, 2095
    HTTPS：443, 2053, 2083, 2087, 2096, 8443

  2. 通知中心/控制中心，加载失败
    Refused to display ‘https://artalk.your_domain/‘ in a frame because it set ‘X-Frame-Options’ to ‘sameorigin’.
    规则 -> 设置 -> 托管转换 -> HTTP响应头，关闭 添加安全性标头 选项。
    因为开启后，CDN 会自动添加标头 x-frame-options: SAMEORIGIN 导致报错。

  3. Artalk 开启了 IP属地 功能，但是没显示
    规则 -> 设置 -> 托管转换 -> HTTP请求头，需要 关闭 删除访问者 IP 标头 选项。
    开启后，cloudflare 会在某些情况下，删除掉 X-Forwarded-For 标头，会影响这个功能。

  4. Artalk 评论上传的图片，加载不出来。请求返回 403 Forbidden
    Scrape Shield 把 Hotlink保护(防盗链保护）关闭。

Artalk-IP属地

这个功能默认是关闭的。
开启后，评论会显示 IP属地，如：广东广州。

如果域名托管到了 CloudFlare 或做了其他 CDN 代理：
Artalk后台，设置 -> 服务器 -> 代理标头名，填入 X-Forwarded-For 。

Artalk后台，设置 -> IP属地：

  • 勾选，启用IP属地展示
  • 数据文件路径，默认是 ./data/ip2region.xdb
    需要到互联网下载 IP属地数据库
    由于是 docker 部署的，之前已经将宿主机的 /data/artalk/data 挂载到容器的 /data 目录。
    将文件 ip2region.xdb 直接上传到 /data/artalk/data/ 上即可。
  • 显示精度，可选 country（国家）、province（省份）、city（城市）。

Artalk-验证码

Artalk后台，设置 -> 验证码：

  • 勾选，启用验证码
  • 选择 CloudFlare 的 Turnstile 服务
    登录 CF 网站，创建组件，将域名添加到组件的主机列表。会返回 站点密钥 和 密钥。（可随时查看）
    复制填入到 SiteKey 和 SecretKey。

Artalk-评论审核

开启本地的 关键字过滤，

  • 勾选，Enabled

词库文件：

  • 词库文件，将需要的敏感词，上传到 /data/artalk/data/ 上，然后添加上来。（注意，映射到容器内路径时 ./data/xxx.txt）
  • 词库文件内容分隔符，建议用 ,（需要和上面的词库文件一致）
  • 替换字符，不建议用 *# 等特殊字符，会造成语法冲突，建议用 X

重新创建容器

从 v2.8.4 版本开始，为了增强安全性，Artalk 不再允许指定 UpGit 的可执行文件路径，请将其加入系统的环境变量中。😃

需要将 upgit 可执行文件，放入到容器内部的 /usr/bin/ 目录下。
可惜的是，容器创建成功后，不能再动态挂载目录了。
需要删除容器，重新创建新的。
但是不用担心，因为对容器的 /data 的数据目录做了挂载，下次创建可以复用，数据不会丢失！

Artalk-邮件通知（用户）

这个功能是指，当用户在站点留下的评论，收到其他人的回复时，自动给用户发送邮件通知，提醒用户。

使用免费的 SMTP 方式，这就需要提供一个邮箱，进行发送邮件的操作。

假设用的邮箱是 lijjiabao9@qq.com，
需要登录 QQ邮箱， （新版UI）在右上角的 账号与安全 -> 设备管理 -> 授权码管理，
生成一个授权码 tokenxxxxxxxxxxx。

Artalk后台，设置 -> 邮件通知

  • 勾选，启用邮件通知
  • 发送方式，smtp
  • 发信人昵称，{{reply_nick}}
  • 发信人地址，lijjiabao9@qq.com
  • 邮件标题，[{{site_name}}] 您收到了来自 @{{reply_nick}} 的回复
  • 邮件模板文件，default

SMTP发送

  • 发送地址，smtp.qq.com
  • 发件端口，587
  • 用户名，lijjiabao9@qq.com
  • 密码，（注意不是邮箱密码，是授权码），tokenxxxxxxxxxxx

Artalk-多元推送（管理员）

这个功能是指，当有用户留下评论时，会给管理员推送消息。

  • 待审评论仍然发送通知，默认为 关闭 状态，当该项设置为 false 时，待审评论不会发送通知
  • 嘈杂模式，默认为 关闭 状态，当该项设置为 false 时，站内仅向管理员回复的消息会发送通知。
    例如「普通用户 A」回复「普通用户 B」，这两个用户之间的通讯不会通知管理员。

默认开启了 邮件通知管理员方式。（需要配置好上面的 邮件通知 功能）
可以切换到其他方式的推送，
需要 取消勾选 邮件通知管理员 的 开启，
并配置其他方式推送（如 Telegram、飞书、钉钉、LINE、Bark、Slack 等）。

这里使用 Telegram 接收消息推送：官方参考教程
在 TG 上通过 @BotFathre 创建好机器人后，得到 api_token
在 TG 上通过 @userinfobot 得到自己 TG 账号的 id，tg_id

Artalk后台，设置 -> 多元推送 -> Telegram

  • 勾选，Enabled
  • ApiToken，填入上述的机器人token，api_token
  • Receivers，填入自己 TG 账号id，tg_id

Artalk-社交登录（用户需注册）

Artalk 默认只需填写昵称和邮箱即可发表评论，无需验证邮箱。

但有时候，我们希望用户能够使用社交账号登录，以减少用户填写信息的时间，或者提高用户信息的真实性，
可以通过启用社交登录来实现这一目的。

启用 邮箱密码登录 功能。(需要配置好 邮件通知 功能，需给用户发送验证码注册)

启用邮箱密码登录后，评论框顶部的昵称邮箱输入框将被隐藏，发送按钮将显示为登录按钮。

用户点击登录按钮后，将会弹出一个登录框，用户可以输入邮箱和密码登录，登录成功后即可发表评论。

并且，用户发表的评论将展示「邮箱已验证」的标识。

用户可以通过邮箱注册账号，Artalk 将向用户邮箱发送一封带有验证码的邮件。
验证码有效期为 10 分钟，验证码发送频率限制为 1 分钟一次。
