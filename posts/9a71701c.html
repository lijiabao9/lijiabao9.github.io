基于 Cloudflare Workers 与 GitHub Actions 自动化部署自建临时邮件系统（Cloud-Mail）

发表于2026-05-31|更新于2026-05-31|DevOps域名管理
|总字数:2.5k|阅读时长:8分钟|浏览量:

在日常开发或测试中，我们经常需要用到临时邮箱（捕获所有邮件）。

利用 Cloudflare 强大的边缘计算能力和完全免费的 D1、KV 数据库，结合 maillab/cloud-mail 开源项目，我们可以极其轻松地搭建一个专属于自己的邮件系统。

本文将详细介绍如何通过 GitHub Actions 实现自动化部署，并完美绕过由于 Cloudflare WAF 拦截导致初始化失败的坑。

⚙️ 第一阶段：Cloudflare 侧的准备工作

1. 创建数据库

登录 Cloudflare 控制台，依次创建以下数据库。

强烈建议命名与项目名称保持一致（即 cloud-mail），这样 GitHub 部署脚本会自动识别，无需额外配置 ID。

  • D1 数据库：命名为 cloud-mail。

    免费额度：总容量 5GB，读 500w 次/天，写 10w 次/天。

  • KV 命名空间：命名为 cloud-mail。

    免费额度：总容量 1GB，读 10w 次/天，写、删、列出 1k 次/天。

(注：如果需要更大容量，可选择切换到 R2 存储，但需要绑定信用卡验证开启，且超出免费额度会扣费，个人使用建议直接使用 KV 即可。)

2. 创建账户 API 令牌

GitHub Actions 需要权限来帮你操作 Cloudflare 资源。

  1. 主页左侧导航栏，点击 管理账户 -> 账户 API 令牌 -> 创建令牌。
  2. 令牌名称：例如 用于 GitHub 上 cloud-mail 部署 Worker 的令牌。
  3. 权限策略：
    • 选择模板里的 Edit Cloudflare workers（默认包含 11 个权限，2 个策略）。
    • 必须手动增加一个策略：选择 D1，并在右侧勾选 读 和 写。
  4. 令牌过期时间：选择 永不过期。
  5. 点击审核并创建，复制保存好生成的 Token（它只出现一次）。

3. 获取账户 ID

回到账号主页，在页面右侧（或点击账户名旁边的三个点），复制并保存 账户 ID（Account ID）。

🛠️ 第二阶段：GitHub 侧的配置与部署

1. Fork 仓库与配置变量

  1. 前往开源项目地址 maillab/cloud-mail 并 Fork 到你的个人仓库。
  2. 进入你 Fork 后的仓库，点击 Settings → Secrets and variables → Actions。

请根据下表配置相应的 Secrets（配置后值不可见） 和 Variables（配置后值可见）：

变量类型 变量名称 示例/说明 备注
Secrets CLOUDFLARE_API_TOKEN 刚才生成的 CF Token 敏感信息，不可见
Secrets CLOUDFLARE_ACCOUNT_ID 刚才复制的 CF 账户 ID 敏感信息，不可见
Variables DOMAIN ["example.com"] 字符串数组格式，你管理的邮箱域名
Variables ADMIN admin@example.com 管理员账号（后缀必须在 DOMAIN 中）
Variables JWT_SECRET MyCustomSecret123 随意输入一串长字符串，不要带特殊字符

⚠️ 避坑指南（关于 CUSTOM_DOMAIN）：
核心脚本在部署时，如果填写了 CUSTOM_DOMAIN，GitHub Actions 会尝试使用 curl 请求去初始化数据库。
由于 GitHub 服务器的 IP 属于数据中心 IP，极易触发 Cloudflare 的 WAF（五秒盾/人机验证），导致返回 HTTP 403 Just a moment… 报错。
因此，请不要配置 CUSTOM_DOMAIN 变量！ 稍后我们在 Cloudflare 的 Worker 中手动绑定自定义域名。

2. 执行手动部署

  1. 点击仓库顶部的 Actions 标签页。
  2. 在左侧选择工作流 🚀 Deploy cloud-mail to Cloudflare Workers。
  3. 点击右侧的 Run workflow 按钮，等待其执行完毕。
  4. 部署成功后，Cloudflare 侧会自动生成名为 cloud-mail 的 Worker，并分配一个原生域名（例如 https://cloud-mail.<你的用户名>.workers.dev）。

🚀 第三阶段：手动初始化与域名绑定

1. 手动初始化数据库（可选）

如果由于是在 GitHub 仓库中配置了 Custom Domain，导致 Actions 工作流的初始化数据库步骤失败，可以手动初始化数据库。

在浏览器中拼接并访问以下 URL：https://<你的Worker原生域名>/api/init/<你的JWT_SECRET>

示例： https://cloud-mail.myusername.workers.dev/api/init/MyCustomSecret123

如果页面返回初始化成功的 JSON 数据，说明数据库结构已准备就绪。

2. 绑定自定义域名（可选）

如果你希望使用自己的域名访问管理后台：

  1. 登录 Cloudflare，进入刚才创建的 cloud-mail Worker 详情页。
  2. 选择 域 (Domain) -> 添加域名（Add Domain），选择要绑定的域名（如 example.com）。
  3. 在 自定义域 (Custom Domains) 下，输入你想使用的子域名（如 cloud-mail.example.com）并确认。

📧 第四阶段：邮件路由与注册

1. 配置 Cloudflare 邮件转发

想要让域名接收入站邮件，需要开启 Cloudflare 的邮件路由功能：

  1. 在 CF 主页，点击进入你的域名详情页。
  2. 选择 电子邮件 -> 电子邮件路由。
  3. 系统会提示你需要配置 3 个 MX 记录和 2 个 TXT 记录，直接点击 添加记录并启用。
  4. 启用后，点击 路由规则 标签页，找到 Catch-all 规则。
  5. 将其状态设置为 激活，点击编辑，将操作设置为 发送到 Worker，目标选择我们创建的 cloud-mail。

2. 注册管理员账号

  1. 访问你的 Worker 域名（原生或自定义域名均可）。
  2. 点击 注册账号，输入你在 GitHub 变量中配置的管理员账号（如 admin@example.com）并设置密码。
  3. 登录成功后，为了安全起见，强烈建议进入 系统设置，关闭 用户注册 功能。

📤 第五阶段：配置邮件发送（整合 Resend）

Cloudflare Workers 默认封禁了 25 端口，不支持直接对外发送邮件。我们需要借助第三方服务 Resend 来实现发件功能。

1. Resend 侧的配置

  1. 注册并登录 Resend。(免费计划额度为每天 100 封，每月 3000 封，足够个人使用)。
  2. 左侧导航栏，选择 Domains -> Add Domain：
    • 输入你的域名
    • 验证方式，建议选择 Auto configure，系统会引导你登录 Cloudflare 并自动添加所需的 DNS 记录
    • 等待其变为 Verified 状态
  3. 左侧导航栏，选择 API Keys → Create API Key：
    • Name: cloud-mail
    • Permission: Full access
    • Domain: All domains
  4. 复制并保存生成的 API Key。

2. 设置状态回调（Webhook）

为了让你的邮件系统能实时得知邮件是否发送成功、被退信或被标记为垃圾邮件，我们需要在 Resend 中配置 Webhook：

在 Resend 左侧导航栏，选择 Webhooks -> Add webhook：

  • Endpoint URL 填写：https://<你的Worker域名>/api/webhooks。
  • Select events to listen 勾选以下事件：
    • email.bounced（被退信）
    • email.complained（被投诉/标记垃圾）
    • email.delivered（成功送达）
    • email.delivery_delayed（发送延迟）
    • email.failed（发送失败）

3. 回填至 Cloud-Mail 后台

  1. 登录你的 cloud-mail 后台管理界面。
  2. 进入 系统设置 -> 邮件设置。
  3. 找到 Resend Token，点击添加，为你对应的域名填入刚才在 Resend 生成的 API Key。

🎯 总结与对比：为什么要用 Cloud-Mail？

看到这里，你可能会想：Cloudflare 官方的邮件路由本来就自带 Catch-all（捕获所有） 并直接转发到我个人邮箱（如 Gmail）的功能。

那我为什么还要大费周折地搭建一个 cloud-mail 系统呢？

相比于单纯的 CF 官方盲目转发，cloud-mail 配合其强大的管理后台，带来了以下降维打击式的优势：

1. 更强大的多渠道分发（支持多邮箱与 Telegram）

  • CF 官方转发： 只能单线联系，或者一条条规则手动维护。
  • Cloud-Mail 处理： 在通过 Catch-all 接收到邮件后，不仅可以将其存储在你的 D1 数据库中供网页端查看，还支持二次转发！ 你可以自由配置将其转发给指定的多个邮箱。
  • 无缝集成 Telegram 机器人： 它原生支持配置 Telegram Bot。 当你的域名收到新邮件时，Telegram 会瞬间弹出消息提醒，你直接在聊天软件里就能完成邮件的即时预览，体验极其流畅。

2. 完美的“收发独立”与多身份隔离管理

如果你使用官方转发，所有的垃圾邮件、注册验证码都会一股脑地塞进你的主力邮箱，久而久之主力邮箱就会被污染。

  • 独立的前缀管理：cloud-mail 会对不同前缀（例如 netflix@你的域名.com、github@你的域名.com）的邮件进行完全独立的收件和发件管理。
  • 完美的业务隔离： 你可以在后台清晰地看到每个前缀收到了哪些信。 在需要对外沟通或注册服务时，能够有效保护主邮箱隐私，还能极其方便地对不同业务进行分类和溯源（谁泄露了你的邮箱，一目了然）。

3. 接入 Resend，实现完美的“真·双向收发”

  • CF 官方转发：只能收信，无法发信。 一旦你需要用这个域名邮箱给别人回复邮件，官方转发就彻底无能为力了。
  • Cloud-Mail ＋ Resend：系统完美接入了 Resend 第三方服务，彻底打破了 Cloudflare 25 端口封禁的限制。你不仅可以作为后台管理员给别人发信，更厉害的是，你还可以针对不同的邮箱前缀，使用各自独立的身份和签名进行回信与主动发送。 每一个前缀都变成了一个功能完整的独立邮箱！

4. 一体化的 Web 客户端与超低成本

  • 拥有一个轻量、美观的 Web UI，你可以随时随地登录它来收发邮件、管理用户、查看日志。

  • 结合前文提到的 Resend 服务，它不仅解决了 Cloudflare 无法发件的痛点，还利用免费额度实现了一套完全由你掌控、多渠道提醒、收发兼备的现代化 Serverless 临时/全功能邮件系统。

  • 在个人低频使用的场景下，这一切依然是完全免费的！
